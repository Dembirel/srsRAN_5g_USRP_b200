# Объяснение кода модуля визуализации eNB

Тема проекта: разработка модуля сбора и визуализации статистических данных базовой станции 4G eNB.

Главная идея реализации: не менять внутренние алгоритмы LTE eNB, а добавить новый listener метрик. eNB уже умеет собирать статистику в структуру `enb_metrics_t`, а `metricshub` раздает эту структуру всем слушателям. Мы добавили еще одного слушателя - `metrics_imgui`, который получает метрики и рисует окно Dear ImGui.

## Какие файлы добавлены

Добавлены два основных файла модуля:

```text
srsenb/hdr/metrics_imgui.h
srsenb/src/metrics_imgui.cc
```

`metrics_imgui.h` - объявление класса, структура состояния GUI, поля KPI, поток, mutex.

`metrics_imgui.cc` - реализация: запуск окна, получение метрик, расчет KPI, рисование ImGui.

Также добавлены исходники Dear ImGui:

```text
third_party/imgui
```

И изменены файлы интеграции:

```text
srsenb/src/main.cc
srsenb/hdr/enb.h
srsenb/src/CMakeLists.txt
configs/zmq/enb.conf
```

## Почему почти не трогали enb.cc

В `enb.cc` не нужно было влезать, потому что srsRAN уже имеет штатный механизм метрик.

Правильный подход:

```text
eNB собирает статистику -> enb_metrics_t -> metricshub -> metrics_imgui -> окно ImGui
```

То есть модуль визуализации не вмешивается в:

- планировщик MAC;
- RRC-процедуры;
- обработку PHY;
- передачу пользовательских пакетов;
- S1AP/GTP-U обмен с EPC.

Это важно для защиты: модуль не ломает LTE-логику, а только отображает уже собранную статистику.

Файл:

```text
srsenb/src/main.cc
```

### 1. Подключен заголовок

```cpp
#include "srsenb/hdr/metrics_imgui.h"
```

Это нужно, чтобы `main.cc` знал про класс `metrics_imgui`.

### 2. Добавлен параметр конфига

В парсер аргументов добавлена строка:

```cpp
("expert.metrics_gui_enable",
 bpo::value<bool>(&args->general.metrics_gui_enable)->default_value(false),
 "Show embedded ImGui metrics window.")
```

Она связывает параметр `metrics_gui_enable` из `enb.conf` с переменной:

```cpp
args.general.metrics_gui_enable
```

### 3. Создается metricshub

В `main()` есть:

```cpp
srsran::metrics_hub<enb_metrics_t> metricshub;
```

`metricshub` - это объект, который периодически берет метрики у eNB и раздает их слушателям.

Дальше он инициализируется:

```cpp
metricshub.init(enb.get(), args.general.metrics_period_secs);
```

Это значит: брать метрики у объекта `enb` с периодом `metrics_period_secs`.

### 4. Добавлен наш listener

```cpp
srsenb::metrics_imgui metrics_gui(args.general.metrics_gui_enable);
if (args.general.metrics_gui_enable) {
  metricshub.add_listener(&metrics_gui);
}
```

Если `metrics_gui_enable = true`, создается объект `metrics_imgui` и добавляется в `metricshub`.

После этого `metricshub` будет вызывать у нашего объекта функцию:

```cpp
set_metrics(...)
```

### 5. Корректная остановка GUI

При завершении eNB добавлено:

```cpp
metrics_gui.stop();
```

Это нужно, чтобы остановить GUI-поток и закрыть окно.

## metrics_imgui.h

Файл:

```text
srsenb/hdr/metrics_imgui.h
```

Главная строка:

```cpp
class metrics_imgui : public srsran::metrics_listener<enb_metrics_t>
```

Это означает, что `metrics_imgui` является слушателем метрик eNB.

`metricshub` может передавать ему структуру:

```cpp
enb_metrics_t
```

Главный публичный метод:

```cpp
void set_metrics(const enb_metrics_t& m, const uint32_t period_usec) override;
```

Он вызывается каждый раз, когда `metricshub` получил новые метрики.

## gui_state_t

Внутри `metrics_imgui` есть структура:

```cpp
struct gui_state_t
```

Она хранит то, что будет показано в окне:

```cpp
float connected_ues;
float dl_mbps;
float ul_mbps;
float dl_tb;
float ul_tb;
float rach_count;
float pci;
float ul_rssi;
float ul_sinr;
float connections;
```

Также есть история для графиков:

```cpp
std::array<float, history_capacity> dl_history;
std::array<float, history_capacity> ul_history;
```

`history_capacity = 90`, то есть хранится до 90 последних значений DL/UL.

## Зачем нужен mutex

В классе есть:

```cpp
std::mutex mutex;
```

Он нужен потому, что есть два потока:

- поток метрик обновляет `state`;
- GUI-поток читает `state` и рисует окно.

Чтобы они не читали и не писали данные одновременно, используется `std::lock_guard<std::mutex>`.

srsenb/src/metrics_imgui.cc

## Конструктор

```cpp
metrics_imgui::metrics_imgui(bool enabled_) : enabled(enabled_)
{
  if (!enabled) {
    return;
  }
  running    = true;
  gui_thread = std::thread(&metrics_imgui::run_gui, this);
}
```

Если GUI выключен, ничего не запускается.

Если GUI включен:

- `running = true`;
- создается поток `gui_thread`;
- поток запускает функцию `run_gui()`.

Важно: GUI работает в отдельном потоке, а не в основном потоке eNB.

## stop()

```cpp
void metrics_imgui::stop()
{
  if (!enabled) {
    return;
  }
  running = false;
  if (gui_thread.joinable()) {
    gui_thread.join();
  }
}
```

Эта функция останавливает GUI-поток.

`running = false` говорит циклу окна завершиться.

`join()` ждет, пока поток реально закончит работу.

## set_metrics()

```cpp
void metrics_imgui::set_metrics(const enb_metrics_t& m, const uint32_t period_usec)
{
  (void)period_usec;
  if (!enabled || !running.load()) {
    return;
  }
  update_snapshot(m);
}
```

Это точка входа для новых метрик.

Эту функцию вызывает `metricshub`.

Если GUI включен и поток работает, вызывается:

```cpp
update_snapshot(m);
```

`period_usec` пока не используется, поэтому написано:

```cpp
(void)period_usec;
```

Это просто чтобы компилятор не ругался на неиспользуемый параметр.

## update_snapshot()

Главная функция расчета KPI:

```cpp
void metrics_imgui::update_snapshot(const enb_metrics_t& m)
```

Сначала блокируется mutex:

```cpp
std::lock_guard<std::mutex> lock(mutex);
```

Дальше берется число подключенных UE:

```cpp
state.connected_ues = static_cast<float>(m.stack.rrc.ues.size());
```

`m.stack.rrc.ues.size()` - количество UE в RRC-метриках.

## Цикл по UE

```cpp
for (size_t i = 0; i < m.stack.rrc.ues.size(); ++i)
```

Проходим по всем UE, которые видит RRC.

Дальше проверяем, что для этого UE есть MAC-метрики:

```cpp
if (i >= m.stack.mac.ues.size()) {
  break;
}
```

Берем MAC-метрики UE:

```cpp
const auto& ue = m.stack.mac.ues[i];
```

## Расчет DL/UL скорости

```cpp
dl_rate_sum += ue.tx_brate / (ue.nof_tti * 1e-3f);
ul_rate_sum += ue.rx_brate / (ue.nof_tti * 1e-3f);
```

Объяснение:

- `tx_brate` - данные, которые eNB передает UE. Это downlink.
- `rx_brate` - данные, которые eNB принимает от UE. Это uplink.
- `nof_tti` - количество TTI, по которому нормируется значение.
- `1e-3f` переводит TTI в секунды, потому что 1 TTI примерно 1 мс.

Потом ниже скорость переводится в Mbps:

```cpp
state.dl_mbps = dl_rate_sum / 1e6f;
state.ul_mbps = ul_rate_sum / 1e6f;
```

## Расчет TB

```cpp
dl_tb_sum += ue.tx_pkts;
ul_tb_sum += ue.rx_pkts;
```

Здесь:

- `tx_pkts` - транспортные блоки, переданные eNB в сторону UE, то есть DL TB;
- `rx_pkts` - транспортные блоки, принятые eNB от UE, то есть UL TB.

## PCI

```cpp
pci = static_cast<float>(ue.pci);
```

PCI - Physical Cell ID.

В GUI он показывается как идентификатор соты.

## RSSI и SINR

```cpp
rssi_sum += m.phy[i].ul.pusch_rssi;
sinr_sum += m.phy[i].ul.pusch_sinr;
radio_samples++;
```

Берутся PHY-метрики uplink:

- `pusch_rssi` - уровень принятого сигнала;
- `pusch_sinr` - отношение сигнал/помеха+шум.

Если есть несколько samples, ниже считается среднее:

```cpp
state.ul_rssi = radio_samples > 0 ? rssi_sum / radio_samples : 0.0f;
state.ul_sinr = radio_samples > 0 ? sinr_sum / radio_samples : 0.0f;
```

## RACH

```cpp
for (const auto& cc : m.stack.mac.cc_info) {
  rach_count_sum += cc.cc_rach_counter;
}
```

RACH - процедура случайного доступа UE к соте.

`cc_info` - информация по компонентным несущим.

`cc_rach_counter` - счетчик RACH.

## Connections

```cpp
if (previous_connected_ues == 0.0f && state.connected_ues > 0.0f) {
  state.connections += 1.0f;
}
```

Счетчик увеличивается, когда раньше UE не было, а теперь появился хотя бы один UE.

Это простой счетчик установления соединения для учебного стенда.

## push_sample()

```cpp
void metrics_imgui::push_sample(float dl_mbps, float ul_mbps)
```

Эта функция добавляет текущие значения DL/UL в историю графика.

Если история еще не заполнена:

```cpp
state.dl_history[state.history_size] = dl_mbps;
state.ul_history[state.history_size] = ul_mbps;
state.history_size++;
```

Если история уже заполнена, массив сдвигается влево:

```cpp
for (size_t i = 1; i < state.dl_history.size(); ++i) {
  state.dl_history[i - 1] = state.dl_history[i];
  state.ul_history[i - 1] = state.ul_history[i];
}
```

А новое значение пишется в конец:

```cpp
state.dl_history.back() = dl_mbps;
state.ul_history.back() = ul_mbps;
```

Так получается простой график последних значений.

## run_gui()

Функция:

```cpp
void metrics_imgui::run_gui()
```

Она отвечает за окно.

## SDL2

Сначала запускается SDL2:

```cpp
SDL_Init(SDL_INIT_VIDEO | SDL_INIT_TIMER)
```

Потом создается окно:

```cpp
SDL_CreateWindow("srsENB ImGui monitor", ...)
```

Потом создается renderer:

```cpp
SDL_CreateRenderer(...)
```

SDL2 отвечает за:

- создание окна;
- обработку событий;
- вывод изображения на экран.

## Dear ImGui

Дальше создается контекст ImGui:

```cpp
ImGui::CreateContext();
ImGui::StyleColorsDark();
```

Подключаются backend-ы:

```cpp
ImGui_ImplSDL2_InitForSDLRenderer(window, renderer);
ImGui_ImplSDLRenderer2_Init(renderer);
```

Это мост между SDL2 и Dear ImGui.

## Главный цикл GUI

```cpp
while (running.load()) {
```

Пока `running = true`, окно работает.

Внутри цикла:

1. Читаются события SDL2.
2. Копируется текущий `state`.
3. Начинается новый кадр ImGui.
4. Рисуются текстовые KPI.
5. Рисуются графики.
6. Кадр выводится на экран.

## Почему используется snapshot

```cpp
gui_state_t snapshot = {};
{
  std::lock_guard<std::mutex> lock(mutex);
  snapshot = state;
}
```

GUI не рисует прямо из `state`.

Он быстро копирует `state` в `snapshot`, отпускает mutex и дальше рисует уже копию.

Это правильно, потому что нельзя долго держать mutex во время рендера окна.

## Рисование текстовых KPI

```cpp
ImGui::Text("Connected UEs : %.0f", snapshot.connected_ues);
ImGui::Text("DL throughput : %.2f Mbps", snapshot.dl_mbps);
ImGui::Text("UL throughput : %.2f Mbps", snapshot.ul_mbps);
ImGui::Text("DL TB         : %.0f", snapshot.dl_tb);
ImGui::Text("UL TB         : %.0f", snapshot.ul_tb);
```

Так выводятся числовые значения.

## Рисование графиков

```cpp
ImGui::PlotLines("DL Mbps", snapshot.dl_history.data(), ...);
ImGui::PlotLines("UL Mbps", snapshot.ul_history.data(), ...);
```

`PlotLines` рисует простую линию по массиву значений.

Мы передаем:

- историю DL;
- историю UL;
- количество заполненных точек;
- минимальное и максимальное значение шкалы;
- размер графика.

## Завершение GUI

В конце `run_gui()` вызывается:

```cpp
ImGui_ImplSDLRenderer2_Shutdown();
ImGui_ImplSDL2_Shutdown();
ImGui::DestroyContext();
SDL_DestroyRenderer(renderer);
SDL_DestroyWindow(window);
SDL_Quit();
```

Это корректно освобождает ресурсы ImGui и SDL2.

## Что говорить преподавателю

Короткий вариант:

> Я добавил в srsENB модуль `metrics_imgui`. Он наследуется от `metrics_listener<enb_metrics_t>`, поэтому подключается к штатному `metricshub`. Когда eNB собирает новую статистику, `metricshub` вызывает `set_metrics()`, а мой модуль выбирает KPI из MAC/RRC/PHY и отображает их в окне Dear ImGui. Окно запускается в отдельном GUI-потоке через SDL2.

Если спросят, почему не изменял `enb.cc`:

> Потому что eNB уже имеет штатный интерфейс метрик. Менять LTE-логику не нужно и даже нежелательно. Правильнее добавить отдельный listener, который только читает статистику.

Если спросят, где именно GUI:

> GUI находится внутри процесса `srsenb`. Он не относится к EPC. В коде это объект `metrics_imgui`, который создает отдельный поток `run_gui()`.

Если спросят, почему Dear ImGui:

> Dear ImGui подходит для инженерных панелей мониторинга. Он позволяет быстро выводить числовые KPI и графики без сложной GUI-архитектуры.

Если спросят, почему SDL2:

> SDL2 используется как backend для окна, событий и рендера. Dear ImGui рисует интерфейс, а SDL2 создает окно и выводит кадры.

Если спросят, почему RSRP/RSRQ не считаются:

> RSRP/RSRQ измеряются на стороне UE и должны приходить в eNB через RRC MeasurementReport. В текущей версии я беру только штатный `enb_metrics_t`, поэтому RSRP/RSRQ оставлены как дальнейшее расширение. Искусственные значения я не вывожу.

Если спросят, почему handover равен 0:

> В текущем ZMQ-стенде одна сотa, поэтому handover фактически не происходит. Счетчик handover имеет смысл добавлять и проверять на стенде с несколькими сотами.

## Общая схема работы

```text
Запуск srsenb
    |
    v
main.cc читает enb.conf
    |
    v
metrics_gui_enable = true
    |
    v
создается metrics_imgui
    |
    v
metrics_imgui запускает GUI-поток run_gui()
    |
    v
metricshub.add_listener(&metrics_gui)
    |
    v
eNB периодически формирует enb_metrics_t
    |
    v
metricshub вызывает metrics_imgui::set_metrics()
    |
    v
update_snapshot() считает KPI
    |
    v
run_gui() рисует окно Dear ImGui
```

## Самая короткая формулировка

Мой модуль - это встроенный в `srsenb` GUI-listener метрик. Он получает `enb_metrics_t` через штатный `metricshub`, считает нужные KPI и отображает их в отдельном окне Dear ImGui/SDL2.
